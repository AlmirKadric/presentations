# GamePad2

What is it

GamePad2 is an in house developed production deployment tool

## But why do we need it?

The Problems

  1. We have no way to check if problems arise in production before showing it to our consumers.
  2. There has always been a slight downtime (~1-5m) when pushing as we have had to wait for processes to restart.
  3. Even when performing a rolling restart there is still a chance the load balancer will forward a request to a stopped machine.
  4. Rollbacks have proved to be a hassle in the past and usually don't work.
  5. The way we used git would cause a plethora of issues, which would cause bad / inconsistent code pushing.
  6. Lack of error detection would cause us to down a machine when doing a bad push, i.e. missing config etc.
  7. Lack of accessibility to team leaders and project managers.
  8. ... And the list goes on ...

So why do we need it:

  1. Ensure true zero downtime deployment.
  2. Ensure we can't perform a bad push.
  3. Ensure we can actually check the deployment as it is on production before our consumers do.
  4. Ensure we have a means to rollback quickly and easily.
  5. A means to provide access to team leaders and project managers.
  6. Ability to test a few possible versions with subsets of players, and test the waters before hard launching.

X-Version Headers

What do you mean?

By providing the X-Version header with your request, you will look at that version folder instead of the current one

That sounds tricky?

We've actually put in place a proxy which will inject this header for you. So you really don't need to do anything for general use cases.

What we expect from you

(Developers)

  1. Tag all releases properly, GamePad2 will only work with tags
  2. Do not perform destructive data migrations, i.e. pushing of x.y.z version means we can't rollback
  3. Make game data versionable, i.e. make them JSON files and put them in the repository
